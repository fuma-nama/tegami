The Maven plugin discovers modules from `pom.xml` files, bumps versions (with parent and `${revision}` inheritance), updates inter-module dependency versions, and publishes with a configurable `mvn deploy`.

```npm
npm install @tegami/maven
```

```ts
import { tegami } from "tegami";
import { maven } from "@tegami/maven";

const paper = tegami({
  plugins: [maven()],
});
```

<Callout type='warn' title="Requirements">

[Maven](https://maven.apache.org) (`mvn`) and a JDK must be available in CI. Deployment configuration (distribution management / the publishing plugin) and credentials live in your `pom.xml` and `settings.xml` — Tegami runs the deploy command, it does not configure the deployment.

</Callout>

## Discovery

Starting from the root `pom.xml`, Tegami recurses through `<modules>` (each `<module>` is a directory with its own pom). Use the `packages` option to include poms not reachable from the root.

Coordinates are `groupId:artifactId` (a module inherits `groupId` and `version` from its `<parent>` when omitted). The Tegami id is `maven:groupId:artifactId`.

**Version resolution**, in order:

1. a literal `<version>` on the project,
2. `${revision}` — resolved from the nearest `<properties><revision>` up the parent chain,
3. inherited from the parent's resolved version.

Tegami tracks which file and element hold the version so bumps edit the right place. Aggregator poms (`<packaging>pom</packaging>`) are included and deployed like any other module.

### Shared / inherited versions

When modules share a version location — a parent's `<version>` or a single `${revision}` property — bumping any member edits that one location, so they release together. If synced modules get different bumps in a single release, the highest wins. Child `<parent><version>` references are updated when the parent is bumped.

## Dependency Bumping

| Dependency scope      | Default dependent bump |
| --------------------- | ---------------------- |
| `compile` / `runtime` | patch                  |
| `test` / `provided`   | none                   |

Inter-module `<dependency>` entries with a literal `<version>` are rewritten to the released version when it changes. `${project.version}` and other property references are left untouched.

## Configuration

```ts
maven({
  publishCommand: ["mvn", "-B", "-ntp", "-DskipTests", "deploy"],
  registry: "https://repo1.maven.org/maven2",
});
```

## Publishing

By default Tegami runs `mvn -B -ntp -DskipTests deploy -pl <module> -am` at the workspace root so inter-module build ordering is handled by Maven. Override the command with `publishCommand`. `-SNAPSHOT` versions are treated as not publishable; customize with `publish`.

Publish status is checked by requesting the deployed `.pom` from `https://repo1.maven.org/maven2` (override with `registry`, or set `registry: false` for private repositories).

<Callout type='info' title="Maven Central">

Publishing to Maven Central through the [Central Publisher Portal](https://central.sonatype.org) requires the `central-publishing-maven-plugin` configured in your pom. Central has a sync delay, so a freshly deployed version may report as pending for a few minutes.

When the [git](/plugins/git), [GitHub](/plugins/github), or [GitLab](/plugins/gitlab) plugin is configured, releases are tagged as `groupId/artifactId@version` (the default `name@version` tag would contain a colon, which git forbids in tag names).

</Callout>

## Limitations

- Only Maven (`pom.xml`) is supported; Gradle is out of scope.
- Version ranges (e.g. `[1.0,2.0)`) in dependency declarations are left untouched.
