The Swift plugin discovers packages from `Package.swift` manifests, bumps versions through git tags, and follows local `.package(path:)` dependencies.

```npm
npm install @tegami/swift
```

```ts
import { tegami } from "tegami";
import { git } from "tegami/plugins/git";
import { swift } from "@tegami/swift";

const paper = tegami({
  plugins: [git(), swift()],
});
```

<Callout type='warn' title="Requirements">

Swift packages publish through git tags, so the Swift plugin requires the [git](/plugins/git), [GitHub](/plugins/github), or [GitLab](/plugins/gitlab) plugin to create tags. No Swift toolchain is required.

</Callout>

## How Swift versioning works

Like Go modules, Swift packages have no version field in their manifest â€” SwiftPM resolves versions from the repository's git tags. Tegami mirrors the [Go plugin](/plugins/go):

- **Current version** is read from existing git tags (or the publish lock at publish time).
- **Next version** is stored in the publish lock during `tegami version`, not in `Package.swift`.
- **Publishing** creates git tags, delegated to the git plugin.

## Discovery

Tegami globs `**/Package.swift` (ignoring `.build`, `node_modules`, and `Pods`). The package name is read from the first `name:` argument of the `Package(` initializer. Manifests where no name can be resolved are skipped.

## Dependency Bumping

Tegami links `.package(path: "../foo")` entries to workspace packages. These local dependents are patch-bumped by default when a dependency is released. `.package(url: ...)` entries are external and ignored â€” there is no version constraint to rewrite for path dependencies.

| Dependency kind          | Default dependent bump |
| ------------------------ | ---------------------- |
| `.package(path:)` (local) | patch                 |

## Configuration

```ts
swift({
  tagPrefix: "v",
  publish: (pkg) => pkg.name !== "InternalTools",
});
```

<auto-type-table type="import('@tegami/swift').SwiftPluginOptions" />

Since Tegami's core `PackageOptions` cannot be extended by external plugins, the `publish` option replaces Go's `packages.go.publish` for disabling publishing per package.

## Publishing

Publishing creates a git tag per package (`{tagPrefix}{version}` at the repo root, `{relativeDir}/{tagPrefix}{version}` for subdirectory packages). When the tag already exists, the package is skipped.

<Callout type='info' title="Subdirectory packages">

SwiftPM resolves a git dependency to the `Package.swift` at the **repository root**, so only the root package is consumable via SwiftPM's git resolution. Tegami still creates `path/version` tags for subdirectory packages to track their releases, but consumers cannot depend on them by URL without repository splitting.

</Callout>

The [Swift Package Registry](https://github.com/swiftlang/swift-package-manager/blob/main/Documentation/PackageRegistry/Registry.md) is not yet supported.
